从LLM中挖因果关系：把生成式模型当观测数据源的pipeline

TL;DR 东京大学 Kato 等人提出了一个五步pipeline，把LLM生成的文档当作"观测数据"，通过事件抽取、规范化、构建共现矩阵，再接因果发现算法，从LLM的知识中提取因果假设图。 背景：LLM知道因果关系吗？ LLM在海量文本上训练，参数里编码了大量关于"什么导致什么"的知识。但这些知识是隐式的——你问它"A会导致B吗"，它可能给你一个看起来合理的回答，但这种one-shot式的询问既不系统也不可靠。 这篇论文（arXiv: 2603.04276, 3月4日）换了个思路：不是直接问LLM因果问题，而是让LLM大量生成某个主题的分析文档，然后从这些文档中系统性地提取因果结构。 核心方法：五步pipeline Step 1: 给定一个主题T（比如"贸易政策对经济的影响"），让LLM生成N篇分析文档（比如100篇）。每篇文档相当于LLM对这个主题的一次"采样"。 Step 2: 用LLM从每篇文档中抽取事件列表。比如"2026年起对主要贸易伙伴重新加征关税"、“保护主义加剧，贸易摩擦增加"这类具体事件。 Step 3: 事件规范化（Canonicalization）。 这是论文最核心的贡献。问题在于：同一个事件在不同文档里可能有完全不同的表述。比如上面两个例子本质上都在说"关税收紧”。如果把原始字符串当作不同变量，下游分析就废了。 做法是：先用embedding模型（text-embedding-3-large）对所有事件编码，L2归一化后用MiniBatchKMeans聚类（最多K_max个簇），然后让LLM给每个簇起一个规范名称（不超过10个词）。这样就把几百个杂乱的事件字符串压缩成了几十个规范变量。 Step 4: 构建文档-事件二值矩阵 Z (N x C)。每行是一篇文档，每列是一个规范事件，Z[i,c]=1表示文档i提到了事件c。这个矩阵就是"观测数据"。 Step 5: 在矩阵Z上跑因果发现算法。论文用了PC算法和其他基于条件独立性检验的方法，输出候选因果图。 关键洞察：为什么这个思路有意思 把LLM的输出当观测数据，这个视角很巧妙。传统做法是直接让LLM回答因果问题（“X导致Y吗？"），但这依赖LLM的单次推理能力，容易受prompt影响。而这篇论文的做法是：让LLM生成大量叙事，然后从叙事的统计结构中提取因果信息。如果LLM在生成文档时倾向于让事件A和事件B共现，而且这种共现在控制了事件C之后消失，那么因果发现算法就能识别出A->C->B这样的结构。 这本质上是在做：把LLM的隐式知识，通过采样+统计的方式，转化为显式的因果图。 局限和作者的诚实 作者明确说了：输出的因果图不保证反映真实世界的因果关系，它只是"LLM认为的因果假设”。这个定位很清醒。LLM的训练数据本身就有各种bias，生成的文档也会继承这些bias。所以这个pipeline的用途更多是hypothesis generation——帮你快速生成一批因果假设，然后你再用真实数据去验证。 和ACIC竞赛的关联 对做因果推断的人来说，这篇论文提供了一个有趣的视角：LLM能否作为因果知识的先验来源？比如在ACIC这类因果效应估计竞赛中，如果你对数据生成过程（DGP）有一些先验假设，可以用类似的方法从LLM中提取候选的DAG结构，再结合数据做结构学习或调整集选择。当然，这还很初步，但方向值得关注。 原文链接: https://arxiv.org/abs/2603.04276

Compute-Optimal QAT: 量化训练的 Scaling Law

(2025) 的结论是花 10% 的训练步数在 QAT 上就够了。但这篇来自 Apple（一作是 EPFL 实习生 Aleksandr Dremov）的论文用大量实验推翻了这个结论。 核心发现：最优 QAT 比例随计算规模增长 作者在 86M 到 2.2B 参数的模型上，跑了从 2.3B 到 1.4T token 的实验，覆盖 1-bit、2-bit、4-bit、6-bit 四种量化位宽。关键发现： 最优 QAT fraction f* 不是常数，而是随着总 token 数增大而增大。计算预算越大，越应该把更多比例分配给 QAT 阶段。 这个最优比例可以用一个简单的统计量来预测：tokens-per-parameter-byte，即总 token 数除以模型参数量再除以量化位宽对应的字节数。这个指标同时编码了模型大小、训练长度和量化难度三个维度的信息。 更大的模型更容易量化（同等 token/param 比率下 QAT 需要更少比例），训练更久的模型更难量化（FP 阶段积累的 pattern 越多，量化后要修正的偏差越大），更低 bit-width 需要更多 QAT 比例。 ...

因果发现的新范式：当众包智慧遇上LLM Agent

TL;DR: 一篇来自Texas A&M和University of Washington的position paper提出，因果结构学习（causal discovery）应该从"纯数据驱动"转向"人机协作"范式——用众包平台收集领域专家的因果知识碎片，用LLM Agent模拟专家来扩展信息获取，最终通过聚合算法拼出单个agent不可能恢复的全局因果图。 背景：因果发现为什么这么难 从观测数据中学习因果结构（通常表示为DAG，有向无环图）是统计学和机器学习的经典难题。难在两个地方：一是搜索空间的组合爆炸——n个变量的可能DAG数量随n超指数增长；二是观测数据本身的固有歧义——很多不同的因果结构可以产生完全相同的条件独立关系（所谓的Markov等价类）。 传统方法分两大流派：基于约束的方法（PC算法、FCI算法）通过条件独立性检验逐步排除不可能的边；基于分数的方法（GES、NOTEARS）通过优化某个评分函数搜索最优DAG。近年来还有基于连续优化的可微方法（比如NOTEARS把离散的无环约束松弛为连续约束），以及基于LLM的方法直接让大模型"猜"因果关系。 但这些方法都有一个共同的天花板：纯数据驱动在变量多、样本有限时效果急剧下降，而人类专家的因果直觉和领域知识一直没有被系统性地整合进来。 核心提议：分布式因果知识的收集与聚合 这篇论文的核心观点是：因果发现本质上是一个分布式决策问题。每个参与者（人类专家或LLM Agent）只掌握关于因果图的局部、不完美的知识——比如一个临床医生可能知道"吸烟导致肺癌"但不清楚基因层面的调控关系，而一个分子生物学家的知识恰好互补。 论文提出的框架包含四个核心模块： 1. 众包平台（Crowdsourcing Platforms） 利用Amazon Mechanical Turk等平台大规模收集非专业人群的因果判断。关键挑战是任务设计——你不能直接让人画DAG，而是要把问题拆解为"A是否会影响B"这样的pairwise判断。论文引用了Zooniverse等公民科学项目的成功经验，认为因果知识的众包收集在技术上已经可行。 2. 专家知识引出（Expert Elicitation） 对于高质量的因果判断，需要设计交互式的知识引出协议。比如使用结构化问卷让专家不仅给出因果方向的判断，还给出置信度。这里涉及到一个经典的认知科学问题：人类对因果关系的直觉判断到底有多可靠？论文引用了Sloman等人的研究，认为人类在局部因果推理上相当准确，但在全局一致性上容易出错——这正好适合"拼图"式的聚合策略。 3. 聚合算法（Aggregation） 拿到一堆来自不同来源的、可能相互矛盾的因果判断后，如何聚合成一个一致的DAG？这是技术上最有挑战的部分。简单的多数投票显然不够，因为需要保证输出是一个合法的DAG（无环）。论文讨论了基于Bayesian aggregation和learning-to-rank的方法，本质上是把每条边的存在与否和方向视为一个需要从嘈杂标注中学习的latent variable。 4. LLM Agent模拟（LLM-based Simulation） 最有意思的部分：用LLM充当"模拟专家"来扩充人类专家的判断。比如让GPT-4扮演"一位有20年经验的心脏病学家"来回答"高血压是否导致左心室肥厚"这类问题。论文承认LLM的因果推理能力有限（本质上是在做pattern matching而非真正的因果推理），但认为在与人类判断聚合后，LLM的贡献可以显著降低所需的人类标注量。 为什么值得关注 这篇论文本身是一篇position paper / survey，没有提出新算法，但它的价值在于提出了一个清晰的研究路线图，把几个原本分散的领域——因果发现、众包计算、LLM for Science——统一到一个框架里。 对做causal inference + ML的人来说，几个值得思考的点： 传统因果发现方法（PC、GES等）的假设在高维场景下很脆弱，融入先验知识几乎是必经之路。但之前的"先验知识"通常是手动编码的硬约束（比如"这条边一定存在"或"一定不存在"），缺乏对不确定性的建模。这篇论文把先验知识建模为带噪声的软约束，更合理。 LLM作为"因果知识的近似oracle"这个定位很务实。不指望LLM做真正的因果推理，只是把它当作一个成本低、覆盖广的信息源，然后用统计方法去校正它的偏差。这个思路在ACIC这类比赛的场景中也有启发——当你有大量协变量但不确定哪些是confounder时，可以用LLM的领域知识做初筛。 聚合算法这部分和meta-learner有直接关联：meta-learner本质上也是在聚合多个弱因果估计。这篇论文讨论的Bayesian aggregation + DAG约束的方法，有可能作为meta-learner的先验结构来使用。 原文链接：https://arxiv.org/abs/2603.02678 作者：Ryan Feng Lin, Yuantao Wei, Huiling Liao, Xiaoning Qian, Shuai Huang 面试关联知识点 Q: 因果发现中PC算法和基于分数方法（如GES）的核心区别是什么？ PC算法是constraint-based：通过条件独立性检验删边，输出Markov等价类（CPDAG）。GES是score-based：在DAG空间中贪心搜索，优化BIC等评分函数。PC对独立性检验的准确性敏感，高维时容易出错；GES搜索空间大但可以整合先验。两者都只能识别到等价类，无法在纯观测数据下确定所有边的方向。 ...
